# Hollowreach — Interaction System Design (draft)

Working notes from design discussion. Not a technical spec yet.

---

## Core principle

Everything is an entity. Every entity can have interactions. All behavior is data-driven via JSON configs. Zero entity-specific logic in code.

## Entity model

- Every entity (NPC, item, door, furniture, anything) has a **JSON config**
- NPCs additionally have personality, backstory, goals, knowledge, likes/dislikes, voice, speech style
- Every entity has a **state** (e.g. locked, unlocked, open, closed, empty, full)
- Every entity has an **interactions list**

## Interactions

An interaction is something the player can do with an entity. Interactions:

- Have a **label** shown to the player (e.g. "Open", "Talk", "Unlock")
- Have **conditions** that control visibility/availability:
  - Entity state (e.g. state == "unlocked")
  - Player has item (item trigger)
  - Flags set from previous interactions
  - Possibly: other entity's state
- Some interactions are **informational only** (e.g. "Locked" — tells you the state, not an action)
- When an item is required, the interaction only **appears** when the player has that item

## Reactions

When a player performs an interaction, a **reaction** happens. A reaction is a bundle of things that can include:

- **Animation** on the entity (door opens, chest lid rises)
- **Sound effect** (door creak, lock click)
- **Dialogue** (NPC speaks — driven by LLM with a context prompt)
- **State change** on the entity (locked → unlocked, closed → open)
- **State change on another entity** (pull lever → gate opens elsewhere, referenced by ID)
- **Item removal** from player inventory (key consumed)
- **Item spawn** (chest gives player an item)
- **Flag set** (for cross-entity conditions)

A reaction may do **all, some, or none** of these. "Examine" might just show text. "Open" might change state + play animation + play sound.

## Item triggers

Some interactions only appear when the player has a specific item. Example:

- Door has "Unlock" interaction
- "Unlock" requires player to have "iron_key"
- Without the key: player just sees "Locked" (informational)
- With the key: "Unlock" appears in the interaction list
- Unlocking consumes the key and changes door state to "unlocked"
- Now "Open" becomes available (requires state == "unlocked")

## NPC specifics

NPCs are entities with extra personality data. Their interactions include:

- "Talk" — always available, drives LLM dialogue
- "Give item" — appears when player is carrying something
- Possibly unique interactions based on NPC role/story

NPC dialogue is generated by LLM using the personality config as system prompt, conversation history as context, and the specific interaction as the trigger.

## NPC autonomy

NPCs are not passive — they act on their own using the **same interaction system** as the player.

### Context areas

The world is divided into **context areas** (not necessarily rooms — any defined region). A context area provides every NPC inside it with:

- All entities in the area and their **full data** (state, interactions, contents)
- Other NPCs present and their visible state
- The player's presence and what they're carrying/doing

### NPC decision loop

Each NPC tick, the LLM receives:

1. **Persona** — the NPC's full JSON config (personality, goals, knowledge, likes, speech style)
2. **Context** — everything in the current area (entities, their states, available interactions)
3. **History** — what has happened recently (conversations, events witnessed)

The LLM then decides what the NPC **does** — not just what it says. It can:

- Use any interaction on any entity (open a chest, drink from a mug, read a book)
- Walk to another area
- Talk to another NPC or the player
- Pick up / give / use items
- Do nothing (idle, observe)

### Same rules as player

NPCs follow the **exact same interaction rules** as the player:

- Same conditions (need a key to unlock a door)
- Same effects (state changes, animations, sounds)
- NPCs have their own **inventory** — they can carry, use, and give items
- NPC personality determines which interactions they choose (Grok drinks, Elara reads, Whisper steals)

### NPC-to-NPC interaction

NPCs can interact with each other through the same system:

- Talk to each other (LLM generates both sides)
- Give items to each other
- React to each other's actions

## NPC memory and growth

NPCs are not static. They **remember and change**:

- Memory of events (who did what, who said what, who gave what)
- Memory changes behavior over time (Grok likes the player after receiving a gift)
- NPCs can form opinions, change goals, learn new information
- NPC persona evolves — it's not a fixed config, it's a living document

Memory is part of the LLM context. The system must manage what fits in the context window — recent events are most important, older memories can be summarized.

## The decision system

One unified system drives all NPC behavior. Each decision cycle:

1. **Build context**: persona + memories + area entities (with states, interactions) + other NPCs + recent history
2. **Feed to LLM**: everything as one prompt
3. **Get one decision**: a single action (go somewhere, use interaction, speak, idle)
4. **Execute**: pathfinding + interaction system + animation + audio

One decision at a time. After execution, new context, new decision. No complex planning — just one step, observe, decide again.

## Entity-driven animation

Entities define **how characters physically behave** near them:

- A bench defines: sit position, sit animation for the character
- A mug defines: drink position, drink animation
- A bed defines: lie down position, sleep animation
- A ladder defines: climb position, climb animation

The character (player or NPC) doesn't know how to sit — the bench tells it. Adding new interactive furniture = adding a JSON config with use-position + animation reference. No code changes.

This is separate from interaction logic (conditions, effects, state). Animation is about the physical "how", interaction is about the logical "what happens".

## Player input: "Say"

The player's primary tool is **free text input**. When near an NPC:

- Interaction list includes **"Say"** — always available for any NPC
- Selecting "Say" opens a text input field
- Player types anything — questions, lies, threats, flattery, requests
- LLM receives the text as player input, NPC responds in character
- Chatterbox speaks the response aloud
- No predefined dialogue options. The player's words are their weapon.

This is the core of the game. The entire "overthrow the king" arc is achieved through conversation.

## The big picture

The game world contains potentially **hundreds or thousands of NPCs** across multiple areas. The player's goal: overthrow the king — achieved entirely through talking, persuading, giving items, building alliances.

NPCs talk to each other, form opinions, spread rumors. A conversation with one NPC can ripple through the entire social network. Loyalties shift. Alliances form and break.

Every NPC remembers. Every word matters.

## NPC resource consumption

With potentially 1000+ NPCs, the context hash system is critical:

- Each NPC stores `last_context_hash`
- On its turn: hash current context → compare → if same, skip
- Most NPCs most of the time: no change → no LLM query
- Only NPCs with dirty context get queried
- Result: maybe 5-10 LLM queries per tick cycle, not 1000

## Items as tools of persuasion

Items are not just objects — they are social currency:

- Give a guard his favorite drink → he likes you more
- Bring evidence of corruption to a noble → they turn against the king
- Steal a letter and show it to the right person → alliances shift

Items + speech = the player's entire toolkit for revolution.
